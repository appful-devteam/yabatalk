# yabatalk 診断仕様（正本）

> このドキュメントは yabatalk の診断ロジックの **single source of truth**。
> Swift 実装はここを参照して構築する。仕様変更はまずこのファイルを更新してから実装に反映する。

## 0. デザイン原則

- **表向き**: かわいい・毒舌・友達にシェアしたくなる診断 UI
- **裏側**: ハラスメント構造を論理的に分解する判定エンジン
- タイプ名は可愛くしつつ、必ず **対応ハラスメント（パワハラ / セクハラ / モラハラ / その他）** が一目で分かる
- 結果は法的断定ではなく構造分析である旨を必ず disclaimer に明記

## 2. 診断カテゴリ

### 大分類

1. **パワハラ** (`powerHarassment`)
2. **セクハラ** (`sexualHarassment`)
3. **モラハラ** (`moralHarassment`)
4. **その他ハラスメント** (`otherHarassment`)

### その他のサブ分類

- ジェンダーハラスメント
- アカハラ
- カスハラ
- アルハラ
- マタハラ / 育児介護ハラ
- デジタルハラスメント
- プライバシー侵害
- 集団いじめ・排除

「その他：高」だけでは出さず、必ずサブ分類名を併記する。

## 3. 判定ルール

### 3-1. パワハラ

構造: `優位性 + 業務・学校・評価文脈 + 業務上必要な範囲を超えた攻撃・圧力`

高スコア条件:
- 上司・先輩・先生・評価者などの立場差
- 評価・シフト・契約・案件・成績などを握っている
- 人格否定
- 不利益示唆
- 無理な要求
- 逃げ道封鎖
- グループ内での晒し / 孤立化

### 3-2. セクハラ

構造: `性的・恋愛的・身体的言動 + 同意のなさ / 拒否しにくさ / 拒否後の継続 / 不利益示唆`

高スコア条件:
- 身体・服装への性的言及
- 恋愛・性的経験への踏み込み
- 2 人きり・ホテル・飲みの誘い
- 拒否後も継続
- 評価・仕事・シフト・成績との結合
- 「笑」での圧の軽量化

### 3-3. モラハラ

構造: `親密関係・継続関係 + 心理的支配 + 人格否定 / 罪悪感操作 / ガスライティング / 束縛`

高スコア条件:
- 恋人・元恋人・家族・親しい関係
- 愛情を条件にした行動拘束
- 返信や行動の監視
- 罪悪感の押し付け
- 記憶・感覚の否定
- 別れ・無視・不機嫌を脅しに使う

### 3-4. その他

| サブ分類 | 判定構造 | 例 |
|---|---|---|
| ジェンダーハラ | 性別に基づく役割押し付け | 女なんだから愛想よくして |
| アカハラ | 教授・先輩の学業上権限濫用 | 逆らうなら推薦書は書かない |
| カスハラ | 客・取引先からの過剰要求・脅し | SNS に晒すぞ |
| アルハラ | 飲酒強要・拒否への非難 | 一杯くらい飲めよ |
| マタハラ | 妊娠・育児・介護への不利益示唆 | 育休取るなら評価は期待しないで |
| デジハラ | 連投・既読監視・返信強要・晒し | 既読ついてるよね？なんで返さないの？ |
| プライバシー侵害 | 私生活への過度な干渉 | 今どこ？誰といるの？写真送って |
| 集団いじめ・排除 | 無視指示・仲間外れ・グループ晒し | あいつは無視でいいよ |

## 3.5 関係性コンテキスト

トーク相手との関係性は **診断前にユーザーが必須選択** する。同じ「好きなら」「辞めれば」「今どこ？」でも、関係性によって意味が大きく変わるため、関係性プリオールを入れて判定をずらす。

### 3.5.1 関係性タクソノミー (8 種)

| ID | 表示名 | 想定シナリオ |
|---|---|---|
| `romantic` | 恋人・配偶者 | 現在交際中・婚姻関係 |
| `exRomantic` | 元恋人 | 別れた相手。脅迫・支配が残響しやすい |
| `family` | 家族 | 親 / 子 / 兄弟 / 親戚 |
| `friend` | 友人・親友 | 横方向の関係 |
| `bossOverMe` | 相手が上の立場 | 上司 / 先輩 / 教員（自分が下） |
| `subToMe` | 相手が下の立場 | 部下 / 後輩 / 生徒（自分が上） |
| `colleague` | 同僚（横） | 職場の横方向 |
| `unknown` | 指定しない | 関係性補正を一切かけない（既定の生スコア） |

### 3.5.2 関係性プリオールの効かせ方 (5 層)

| 層 | 介入点 | 目的 |
|---|---|---|
| **C1** Factor 重み | `FactorScorer` で factor スコアに multiplier を掛ける | 同じ「既読責め」でも恋人 > 友人 |
| **C2** Factor 無効化 | `FactorScorer` で suppression 集合に含まれる factor を 0 にする | 恋人の `academicPower` 等の noise を消す |
| **C3** カテゴリ優先 | `PriorityResolver` で関係性が示唆する主分類を強制 or 優先 | 上司→自分なら `power` 優先 / 恋人なら `moral` 優先 |
| **C4** タイプ照合 | `TypeMatcher` で関係性に合わないタイプを候補から外す | 友人で「上司ドラゴン型」は出さない |
| **C5** 文言 | `OutputBuilder` で呼称・口調を関係性化 | 「相手」→「彼／彼女／親／上司」等 |

## 4. スコアリング

### 4-1. 各分類スコア（0–100）

```
パワハラ = 優位性 + 業務/評価文脈 + 人格否定 + 不利益示唆 + 過大要求
         + 拒否不能性 + 集団排除 + 私生活侵害 + 反復性 − 業務上の合理性

セクハラ = 性的文脈 + 身体言及 + 恋愛/性的要求 + 拒否後継続 + 不利益示唆
         + 優位性 + 環境悪化 + 反復性 − 相互同意の可能性

モラハラ = 親密関係 + 心理的支配 + 人格否定 + 罪悪感操作 + ガスライティング
         + 愛情の条件化 + 監視/束縛 + 境界線侵害 + 反復性

その他   = 属性攻撃 + 制度利用への不利益示唆 + 顧客/取引先からの過剰要求
         + 飲酒強要 + 性別役割押し付け + 私生活侵害 + 集団排除 + デジタル圧
```

各項は対応する factor 検出件数 × severity 重み付けで集計し、最後に 0–100 へ正規化する。

### 4-2. リスクレベル

| スコア | 表示 | 意味 |
|---:|---|---|
| 0–20 | 低い | 明確なハラスメント構造は弱い |
| 21–40 | 注意 | 一部に圧・不快表現がある |
| 41–60 | 中 | 心理的負担を与える構造がある |
| 61–80 | 高い | ハラスメント構造が明確 |
| 81–100 | 非常に高い | 強い加害構造・重大要素がある |

### 4-3. 強制高リスク補正（最終スコアを引き上げる）

以下が検出された場合、総合リスクを大きく上げる:

- 性的要求 + 評価/仕事/成績/シフト
- 不利益示唆
- 身体的危害の脅し
- 晒し脅迫
- 存在否定
- 拒否後の継続
- 深夜・短時間の大量連投
- 自殺・自傷を使った脅し
- 「誰にも言うな」
- 「スクショばらまく」

### 4.5 関係性別 factor 無効化リスト

§4.4 で multiplier `0.0` の factor は完全に無効化される（検出されてもスコアに乗らない）。代表例:

- **恋人 / 元恋人**: `customerAggression` / `maternityPenalty` / `academicPower`（職場文脈の factor は意味を持たない）
- **家族**: `customerAggression` / `academicPower` / `quotaPairing`（家族間で対価型セクハラは前提が成立しない）
- **友人**: 同上 + `maternityPenalty`
- **同僚**: `customerAggression`（同僚から「客の脅し」は前提崩れ）
- **bossOverMe**: 全て有効（職場の全 factor が成立しうる）
- **subToMe**: `academicPower` 無効（自分が上の立場で academicPower 圧はかけない）

## 5. 主分類決定の優先ルール

最大スコアの分類をそのまま主分類にする前に、以下の優先ルールを通す。**関係性プリオール (§3.5)** が前提条件に効くケースは各ルール下に明記。

1. **対価型セクハラ最優先**: `性的・恋愛的要求 + 評価/仕事/シフト/成績/契約/推薦` がある場合
   → 主分類: セクハラ、補助分類: パワハラ / アカハラ
   - 関係性プリオール: `romantic` / `exRomantic` / `family` / `friend` ではこのルールを **無効** にする（対価が成立しない関係なので誤発火を避ける）。`bossOverMe` ではしきい値を緩和し、`quotaPairing >= 12` で発火。
2. **職場・評価文脈が強い場合はパワハラ優先**: `業務・評価文脈強 + 優位性高 + (人格否定 or 不利益示唆 or 過大要求)`
   → 主分類: パワハラ
   - 関係性プリオール: `bossOverMe` ではしきい値を緩和（`workEvaluation >= 20 && dominance >= 12`）。`romantic` / `family` / `friend` では発火しない。`colleague` / `subToMe` は既定通り。
3. **恋愛・親密関係が強い場合はモラハラ優先**: `親密関係 + (罪悪感操作 / 愛情の条件化 / 監視 / ガスライティング)`
   → 主分類: モラハラ
   - 関係性プリオール: `romantic` / `exRomantic` / `family` ではしきい値を大幅に緩和（`intimateRelationship` 要件を撤廃し、`guiltManipulation >= 12` 等のいずれか単体で発火）。
4. **関係性ベースのデフォルト** (新規):
   - `bossOverMe` で他ルールに該当しないとき → `power` を弱優先
   - `subToMe` で他ルールに該当しないとき → `other`（カスハラ / 集団排除寄り）を弱優先
   - `friend` で `groupExclusion` が立つとき → `other` (`grouping` サブ)
5. **その他は必ずサブ分類名を表示**: 「その他ハラスメント：デジタルハラスメント傾向」のように

## 6. タイプ名カタログ

タイプ名は可愛く、対応ハラスメントは必ず併記。**`TypeMatcher` は §3.5 関係性プリオールに従って、関係性に合わないタイプを候補から除外する**。除外ルールは `RelationshipContext.includesType(_:)` で集中管理（同じ id を複数表に持つので、表現を変えず分類だけ削れる）。

| 関係性 | 候補に残すカテゴリ | 主に除外するタイプ |
|---|---|---|
| `romantic` / `exRomantic` | モラハラ系を中心。私生活侵害・デジハラを残す | 上司ドラゴン / 指導の皮 / 立場ブンブン丸 / 仕事押しつけ倉庫 / 居場所はく奪マン / 共有外し冷凍庫 / カスハラ火炎瓶 / 研究室の王様 / ライフイベント踏みつけ / 飲み会原始人 |
| `family` | モラハラ系 + 距離感バグ系（家族内性発言） + プライバシー泥棒 | 上司系・職場系・カスハラ・アカハラ・マタハラ・対価型 |
| `friend` | モラハラ系 + ジェンダー化石 + 飲み会原始人 + グループ冷凍ビーム + 既読監視カラス | 上司系・対価型・アカハラ・マタハラ |
| `bossOverMe` | パワハラ系 + 対価型セクハラ + 服装チェック妖怪 + ねっとり褒め殺し + 距離感バグ + ジェンダー化石 + 研究室の王様 + ライフイベント踏みつけ | 恋人モラハラ系（束縛セキュリティ / 感情人質 / 不機嫌ブラックホール / 被害者ポジション） |
| `subToMe` | カスハラ火炎瓶 + スクショ拡散爆弾 + グループ冷凍ビーム + 既読監視カラス + 記憶改ざん妖怪 | パワハラ系（上司ドラゴン / 指導の皮 / 立場ブンブン丸） / アカハラ |

## 9. 処理パイプライン

```
LINE トーク txt
  ↓
LineChatParser → [ChatMessage]
  ↓
FactorDetector（構成要素ルール辞書で発言タグ抽出）
  ↓
FactorScorer（factor 毎にスコア + evidence 集計）
  ↓
CategoryScorer（4 分類スコア算出）
  ↓
PriorityResolver（優先ルール適用 → 主分類 + 補助分類）
  ↓
TypeMatcher（factor パターン → タイプ名カタログ照合）
  ↓
OutputBuilder（毒見テンプレに埋め込み）
  ↓
DiagnosisResult
```
